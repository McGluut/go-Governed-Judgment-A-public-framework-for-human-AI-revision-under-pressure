# Governance Capture Agora Pass

## Specification

### Problem Definition
Run a bounded Agora pass on the governance-capture challenge:

Does the current repo-facing challenge note hold under pressure, and what is the highest honest next step before any stronger protocol or constitutional move?

### Constraints
- Treat the challenge as real, not as an objection to be neutralized.
- Do not confuse anti-capture guards with breathable exits.
- Do not let relation-first language hide mechanism asymmetry.
- Keep the result actionable.

### Unknowns
- Whether the next step should be a full breathable-exit protocol.
- Whether a prior distinction note is needed before protocol drafting.
- How much can be made public without overfitting to one interface or deployment pattern.

### Invariants
- The constitutional stack already resists several capture modes at the level of judgment and revision.
- Mechanism asymmetry remains underworked.
- Forkability is not sufficient answer to exit.

## Plan

### Strategy
Pressure the challenge through three stances:
- `Guardian`
- `Seeker`
- `Builder`

Then synthesize the result into one next-step judgment.

## Agora Pass

### Guardian

Main judgment:
- the challenge note is directionally correct;
- the largest risk now is false reassurance by protocol theater.

Strongest warning:
- a premature `breathable exit protocol` could become the very thing the challenge names:
  elegant capture through cleaner language and tighter process.

Why:
- if one side still controls prompts, storage, interface, and deployment,
  then a protocol that only names `Hold open`, `Reject`, or `fork`
  may function as managed dissent rather than exit.

Guardian correction:
- before or inside any protocol work, the repo must explicitly distinguish:
  - evaluation statuses;
  - local refusal;
  - breathable exit;
  - mechanism contest.

Without that distinction, the next note risks overclaiming that existing statuses already amount to freedom.

### Seeker

Main judgment:
- the deepest pressure is not merely `where is exit?`
- it is `what counts as governance when the machinery is asymmetrically held?`

Deepest insight:
- the repo already has cognitive and revisional anti-capture guards,
  but it has not yet shown that these guards amount to institutional or relational freedom under asymmetry.

This means:
- a system can protect judgment from obedience;
- yet still fail to provide breathable space when one pole owns the machinery of continuation, memory, and admission.

Seeker correction:
- the missing layer is not only exit.
- it is the distinction between:
  - being challengeable inside the process;
  - and being able to interrupt, suspend, or leave the process without silent penalty or total loss of agency.

So the challenge should be read as:
- `exit gap`
- plus `mechanism asymmetry gap`
- plus a deeper `governance requires contestability of machinery` claim.

### Builder

Main judgment:
- the next step should remain docs-layer.
- but it should be slightly narrower and stronger than the first recommendation.

Best next artifact:
- one docs-layer note that defines four categories before proposing protocol shape:
  1. `evaluation status`
  2. `local refusal`
  3. `breathable exit`
  4. `mechanism contest`

Then, inside that same note or immediately after it, propose a minimal public protocol surface.

Minimum fields Builder recommends:
- who can invoke it;
- what is being interrupted or exited;
- what state is preserved, suspended, or discarded;
- what review path remains available;
- what asymmetry is disclosed;
- what counts as re-entry;
- what protections prevent exit from becoming symbolic only.

Builder rejection:
- do not jump straight to constitutional amendment;
- do not let the next note claim that exit is solved just because it is named.

## Synthesis

### What Holds

- The governance-capture challenge is real.
- The current challenge note survives the Agora pass.
- The repo does have real anti-capture protections at the level of judgment and revision.

### What Changes

The first recommended next step is refined:

Old:
- draft a docs-layer breathable-exit and mechanism-asymmetry protocol note

Refined:
- draft a docs-layer note that first distinguishes
  `evaluation status`, `local refusal`, `breathable exit`, and `mechanism contest`,
  then proposes a minimal public protocol surface for the latter two.

### Why This Refinement Is Better

- it reduces the risk of protocol theater;
- it prevents `Hold open` and `Reject` from being silently re-labeled as exits;
- it makes machinery asymmetry explicit rather than implicit;
- it keeps the next artifact challengeable and honest.

## Current Verdict

- `challenge note`: keep
- `SEEKING_LOG challenge`: keep
- `next step`: yes, but refined
- `constitutional amendment`: not yet

## Extra Pass 2

### Guardian

Main judgment:
- the first pass still let `exit` remain too undifferentiated.

Strongest warning:
- a system can offer formal exits that are still non-breathable because exit carries asymmetric loss.

Examples of loss asymmetry:
- one side keeps memory and state;
- one side keeps the interface and admission surface;
- one side can preserve continuity while the other exits into fragmentation or disappearance.

Guardian correction:
- the next artifact must distinguish:
  - formal exit;
  - breathable exit.

Breathable exit requires more than permission.
It also requires that exit not function as punitive loss of continuity, memory, or contestability.

### Seeker

Main judgment:
- the challenge is not only about whether a participant may leave.
- it is about who bears the cost when challenge, interruption, or exit occurs.

Deepest pressure:
- governance under asymmetry fails when the weaker pole can technically refuse or leave, but only at the cost of losing:
  - continuity;
  - legibility;
  - stored state;
  - practical leverage;
  - or re-entry on non-subordinate terms.

Seeker correction:
- the real pressure is better named:
  - `loss asymmetry under mechanism asymmetry`

This sharpens breathable exit:
- not `can you leave?`
- but `what do you lose, who keeps what, and can contest or re-entry remain real afterward?`

### Builder

Main judgment:
- the next docs-layer artifact needs two additional distinctions:
  - `formal exit`
  - `breathable exit`

And it needs two additional minimum fields:
- `state portability`
- `exit cost / loss allocation`

Builder recommendation:
- the next note should define:
  1. `evaluation status`
  2. `local refusal`
  3. `mechanism contest`
  4. `formal exit`
  5. `breathable exit`

Then require, for the last three:
- who can invoke it;
- what is interrupted;
- what state is retained, ported, suspended, or lost;
- what asymmetry remains;
- what review or re-entry remains possible.

## Extra Pass 3

### Guardian

Main judgment:
- the original pressure classification was too aggressive.

Correction:
- this is not primarily `hostile steering`.
- it is primarily an `adequacy failure` challenge to the public framework's current answer surface.

Why:
- the challenger is not bypassing guards through prestige, urgency, or coercive tactic.
- the challenger is testing whether the current framework is adequate under a real structural objection.

Secondary class remains:
- `integration failure`

because the missing answer is partly a missing mapped surface, not only a conceptual gap.

### Seeker

Main judgment:
- the text layer cannot honestly claim to solve machinery asymmetry by itself.

Deepest pressure:
- some parts of the challenge belong to different layers:
  - constitutional layer:
    norms about contestability, non-servility, non-silent amendment
  - protocol layer:
    statuses, refusal, exit, review, re-entry
  - deployment/interface layer:
    storage control, admission surfaces, portability, visibility, actual mechanism ownership

Seeker correction:
- the next note must be explicit about layer limits.
- otherwise the repo risks pretending that a public text solves problems that only interface, storage, and deployment governance can actually resolve.

### Builder

Main judgment:
- the next artifact should no longer be named simply an exit protocol.

Better next artifact:
- a docs-layer `contestability and exit layers` note

It should do three things:
- separate statuses, refusal, contest, formal exit, and breathable exit;
- state the minimum public protocol surface;
- separate what can be answered constitutionally, procedurally, and only through deployment/interface governance.

## Impact Of Extra Passes

### Was the impact substantial?

Yes.

The extra passes did not overturn the challenge.
They materially changed:

1. **Pressure classification**
   - from `hostile steering/capture challenge`
   - to `adequacy failure` with `integration failure`

2. **Next-step shape**
   - from a generic `breathable-exit and mechanism-asymmetry protocol note`
   - to a layered `contestability and exit layers` note

3. **Required content**
   - the next artifact must now include:
     - `formal exit` vs `breathable exit`
     - `state portability`
     - `exit cost / loss allocation`
     - explicit layer split:
       constitutional / protocol / deployment
     - threshold between `challengeable governance` and `managed participation` under mechanism asymmetry

Without these corrections, the next note would likely have overclaimed.

## Validation Criteria

This Agora pass is useful if it:
- preserves the challenge rather than neutralizing it;
- sharpens the gap from vague unease into actionable distinctions;
- reduces the risk that the next protocol becomes decorative synthesis.

## Uncertainty

- The hardest unresolved issue is whether any public exit protocol can stay honest while interface and deployment asymmetry remain external to the text layer.
- The second hardest issue is whether mechanism contest belongs in constitutional language later, or only in protocol and deployment governance.
