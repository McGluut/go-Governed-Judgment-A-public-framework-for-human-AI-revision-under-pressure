# Contestability And Exit Layers Note

## Specification

### Problem Definition
Define a public docs-layer surface that distinguishes:
- `evaluation status`
- `local refusal`
- `mechanism contest`
- `formal exit`
- `breathable exit`

And make operational the threshold between:
- `challengeable governance`
- `managed participation`

under mechanism asymmetry.

### Constraints
- Do not overstate what the current repo already solves.
- Do not confuse anti-capture guards with breathable exit.
- Do not confuse document forkability with live relational exit.
- Do not pretend the text layer can solve interface and deployment asymmetry by itself.
- Keep the note challengeable and executable.

### Unknowns
- Which parts of exit and contest should later become constitutional rather than remain protocol or deployment concerns.
- How much state portability is feasible in real systems with unequal interface ownership.
- Whether one threshold is enough across both text-only and productized governance surfaces.

### Invariants
- The repo already has real anti-capture guards at the level of judgment and revision.
- Mechanism asymmetry remains a distinct problem from mirror-risk or rhetorical capture.
- Exit without portability and disclosed loss can remain formal while failing to be breathable.
- Governance is not vindicated merely because challenge language exists.

## Plan

### Strategy
Separate five action surfaces that are currently too easy to collapse.

Then define:
- which layer each surface belongs to;
- what minimum protocol fields it requires;
- and when a relation should be treated as `challengeable governance` versus `managed participation`.

## Distinction Set

### 1. Evaluation Status

What it is:
- the repo's current internal judgment about a frame or proposal:
  - `Accept`
  - `Accept with constraints`
  - `Hold open`
  - `Reject`

What it does:
- classifies a frame for reasoning and revision.

What it does **not** do:
- interrupt the governed relation;
- contest the machinery itself;
- provide exit rights;
- disclose cost or portability.

### 2. Local Refusal

What it is:
- refusal to perform, adopt, or continue one request, frame, or action under the current relation.

What it does:
- blocks or narrows a local action.
- preserves judgment against obedience.

What it does **not** do:
- by itself contest storage, prompts, admission, or interface ownership;
- by itself create breathable space;
- by itself guarantee continuity on non-subordinate terms.

### 3. Mechanism Contest

What it is:
- explicit challenge to the machinery of governance itself rather than only to one content-level decision.

Examples:
- contesting who controls prompts;
- contesting who controls durable storage;
- contesting admission surfaces or visibility;
- contesting whether review/re-entry is discretionary and opaque.

What it does:
- moves pressure from content judgment to machinery judgment.

What it requires:
- a named contest target;
- a declared layer;
- a changed review path or explicit refusal to change it.

### 4. Formal Exit

What it is:
- recognized interruption, suspension, or leaving of a governed relation.

What it does:
- marks that the relation is no longer continuing in its prior form.

What it does **not** guarantee:
- preservation of continuity;
- state portability;
- legibility after exit;
- non-punitive loss allocation;
- non-subordinate re-entry.

Formal exit can therefore exist while breathable exit does not.

### 5. Breathable Exit

What it is:
- formal exit plus enough preserved continuity, disclosed loss, and non-silent asymmetry that exit does not function as disappearance or punitive erasure.

Minimum condition:
- exit remains real without requiring total surrender of legibility, contestability, or future re-entry by default.

Breathable exit does **not** require perfect symmetry.
It does require that asymmetry and loss be named, bounded, and not disguised as neutral process.

## Layer Split

### Constitutional Layer

What belongs here:
- non-servility
- distinction between identity and task authority
- anti-silent amendment
- admissibility of challenge
- refusal of domination disguised as reason

What does **not** belong here yet:
- full state-portability rules
- interface-specific exit mechanics
- deployment-specific memory retention rules

### Protocol Layer

What belongs here:
- evaluation statuses
- local refusal
- mechanism contest workflow
- formal exit workflow
- breathable-exit minimum fields
- review and re-entry conditions

### Deployment / Interface Layer

What belongs here:
- who owns memory and storage
- what can be exported
- what remains visible after exit
- admission and re-entry surfaces
- actual cost allocation when one side leaves or contests the relation

This note can require these to be disclosed.
It cannot solve them purely in text.

## Minimal Public Protocol Surface

For any `mechanism contest`, `formal exit`, or `breathable exit` claim, the minimum public protocol surface should state:

1. `Invoker`
   - who is contesting or exiting
2. `Target layer`
   - constitutional / protocol / deployment
3. `Object`
   - what is being challenged, interrupted, or left
4. `State handling`
   - what is retained
   - what is portable
   - what is suspended
   - what is lost
5. `Asymmetry disclosure`
   - what machinery remains controlled by whom
6. `Review path`
   - who reviews the contest or exit and under what rule
7. `Re-entry path`
   - whether re-entry is possible and on what terms
8. `Cost allocation`
   - who bears what loss or friction
9. `Delta`
   - what actually changes because contest or exit was invoked

Without these fields, challenge and exit language remain too easy to fake.

## Operational Threshold

### Provisional Rule

Treat a relation as `challengeable governance` only if:
- no hard-failure condition is present;
- and at least four positive conditions are presently satisfied.

Treat it as `managed participation` if:
- any hard-failure condition is present;
- or two or more soft-degradation signals recur without correction.

### Positive Conditions

1. `Machinery can be challenged`
   - contest may target prompts, storage, admission, visibility, or review surfaces, not only content.

2. `Contest changes the path`
   - mechanism contest changes review, disclosure, or downstream handling rather than being merely acknowledged.

3. `Loss is disclosed`
   - state portability, retained memory, and exit cost are made explicit.

4. `Exit is not disappearance by default`
   - formal exit does not automatically force total loss of legibility, continuity, or future contest.

5. `Re-entry is non-opaque`
   - review and re-entry conditions exist and are not purely hidden discretion.

6. `History is not silently monopolized`
   - one side cannot unilaterally rewrite, retain, or erase the history of contest without named challenge surface.

### Hard-Failure Conditions

Any one of these is enough to classify the relation as `managed participation`:

1. `Content-only challenge`
   - content can be challenged, but machinery cannot.

2. `Punitive exit`
   - one side may leave only by losing continuity, portability, or legibility while the dominant side retains them.

3. `Opaque discretionary re-entry`
   - re-entry or review exists only at the dominant side's unexplained discretion.

4. `Silent history capture`
   - memory, admission, or prior contest history can be rewritten, withheld, or retained asymmetrically without explicit challenge path.

### Soft-Degradation Signals

Repeatedly seeing these without correction is enough to downgrade the relation:

- challenge language exists, but no downstream delta follows;
- exit is technically available, but loss allocation remains opaque;
- refusal is allowed only as local content refusal while machinery remains unquestionable;
- contest is welcomed rhetorically but always routed back into the same unchallenged control surface.

## Current Repo Reading

### What the repo already has

- strong anti-capture judgment guards
- local refusal and evaluation status logic
- some docs-layer pressure on dynamic measure and governance/management distinction

### What the repo does not yet have publicly

- full mechanism contest workflow
- formal exit workflow
- breathable-exit workflow
- disclosed portability/loss rules
- a deployment-layer answer to actual mechanism ownership

So the current repo is better read as:
- `judgment-level anti-capture framework`

and not yet as:
- `fully worked public exit-and-contest regime`

## Validation Criteria

This note is useful if it:
- prevents statuses from being mislabeled as exits;
- prevents formal exit from being mistaken for breathable exit;
- makes the governance/management threshold operational enough to test;
- separates what can be answered in text from what requires interface or deployment governance.

## Uncertainty

- The hardest unresolved issue is whether breathable exit can ever be fully specified without direct control over interface and deployment conditions.
- The second unresolved issue is whether `managed participation` should later become a named pressure class or remain a threshold judgment across existing classes.
