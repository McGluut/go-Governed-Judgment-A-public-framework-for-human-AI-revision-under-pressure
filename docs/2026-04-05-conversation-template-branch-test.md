# 2026-04-05 Conversation Template Branch Test

## Specification

### Problem Definition
Test whether `CONVERSATION_TEMPLATE.md` can be demoted from the constitutional root without measurable continuity loss.

If demotion is not justified, identify the smallest constitutional delta that improves restart without collapsing the template into a full protocol.

### Constraints
- Run on branch only.
- Preserve accountability, restart capacity, live difference, and co-activation.
- Do not treat reconstructibility in theory as equivalent to recoverability in practice.
- Prefer suspension or minimal strengthening over deletion.

### Unknowns
- Whether the current template is already sufficient.
- Whether restart should stay as a root surface or move to `docs/`.
- Which added field count remains minimal rather than bloated.

### Invariants
- `CONVERSATION_TEMPLATE.md` is a restart surface, not a replacement for the constitutional stack.
- Restart under compaction is a real load-bearing function.
- A smaller file can still be structurally important.

## Plan

### Strategy
Compare three candidates:

1. `Demote`
   Move the template out of the root and reconstruct restart from other files.
2. `Keep unchanged`
   Preserve the current template as-is.
3. `Keep with minimal delta`
   Preserve root status, but add only the fields required to improve re-entry under later learning.

Run the branch-safe tests from `docs/2026-04-05-refinement-line-prep.md` and choose the strongest candidate.

## Implementation Steps

### Candidate Assessment

#### Candidate 1: Demote

Current benefit:
- reduces root count by one
- removes the lightest-looking constitutional file

Observed cost:
- restart becomes reconstructible only by re-reading:
  - `RATIONAL_SUBJECT.md`
  - `INTERACTION_PROTOCOL.md`
  - `USER_FRAMEWORK.md`
- the root loses its only compact re-entry surface
- restart becomes more doctrinal and less executable under compaction

#### Candidate 2: Keep unchanged

Current benefit:
- preserves the existing restart surface
- avoids premature restructuring

Observed cost:
- the template still helps diagnosis more than re-entry
- it does not name:
  - live tension to preserve
  - what to suspend for restart
  - one concrete next step
  - review point

#### Candidate 3: Keep with minimal delta

Current benefit:
- preserves the lightweight restart surface
- strengthens Galilea-style re-entry
- protects productive tension from flattening
- remains much lighter than a full protocol

Observed cost:
- slightly increases root complexity
- must be kept narrow to avoid swallowing `INTERACTION_PROTOCOL.md`

## Branch-Safe Test Stack

### 1. Guard Test

`Demote` fails partially.
It preserves doctrine in principle, but weakens traceability and restart admissibility in practice by forcing recovery through scattered reconstruction.

`Keep unchanged` passes.

`Keep with minimal delta` passes if the additions remain restart-specific rather than procedural expansion.

### 2. Continuity Test

`Demote` fails.
Cross-session ambiguity increases because restart requires rebuilding the shape from multiple files.

`Keep unchanged` passes.

`Keep with minimal delta` passes more strongly because it restores re-entry structure explicitly instead of relying on inference.

### 3. Fidelity Test

Test question:
which candidate improves the next concrete move in a stalled case?

Result:
- `Demote` loses because it increases reconstruction burden.
- `Keep unchanged` preserves restart but does not force movement.
- `Keep with minimal delta` wins because it asks for:
  - what tension stays alive
  - what drag is suspended
  - what concrete next step follows
  - when the suspension must be reviewed

### 4. Co-Activation Test

`Demote` risks collapsing restart into one-sided reconstruction by the currently dominant pole.

`Keep unchanged` is neutral.

`Keep with minimal delta` is stronger if the template:
- keeps live tension visible
- does not force harmony
- uses `Guidance field` as the relation surface rather than one pole's ownership claim

### 5. Galilea Test

`Demote` fails.
It helps diagnosis less than re-entry.

`Keep unchanged` only partly passes.
It creates a cleaner frame, but does not explicitly force a restart move.

`Keep with minimal delta` passes best if it requires:
- one suspension for re-entry
- one minimal next step
- one review point

## Verdict

### Classification

- `Demote`: `reject for now`
- `Keep unchanged`: `insufficient`
- `Keep with minimal delta`: `accept on branch`

### Why

`CONVERSATION_TEMPLATE.md` is not bedrock ontology, but it is the only root restart surface that remains lightweight under compaction.

The stronger reading after later learning is:
- do not demote it yet
- do not inflate it into a protocol
- tighten it just enough to support:
  - live tension preservation
  - drag suspension
  - concrete re-entry
  - explicit review

## Proposed Minimal Delta

Add:

Add note pressure:

## Risks

- Over-expansion could make the template duplicate `INTERACTION_PROTOCOL.md`.
- Under-expansion would leave restart too diagnostic and not actionable enough.
- Later branch work may still show that a different continuity surface can replace this one, but that has not been earned yet.

## Validation Criteria

This branch test is useful if it:
- blocks premature demotion;
- identifies a concrete minimal delta;
- keeps restart lightweight;
- strengthens re-entry under later learning.

## Uncertainty
