# 2026-04-05 User Framework Branch Test

## Specification

### Problem Definition
Test whether `USER_FRAMEWORK.md` is the right constitutional vessel for continuity.

If the current vessel is too rigid, identify the smallest branch-safe delta that strengthens continuity without:
- collapsing the active guidance field into one pole;
- duplicating `INTERACTION_PROTOCOL.md` or `VALUE_SIGNAL_LOOP.md`;
- weakening accountability or revisability.

### Constraints
- Run on branch only.
- Preserve user-side continuity without overfixing the user as permanent measure.
- Preserve relation-first structure.
- Keep the file narrow enough to remain a continuity surface, not a general protocol.
- Prefer minimal delta over rename, merge, or deletion unless stronger evidence forces it.

### Unknowns
- Whether the file should remain standalone in the constitutional root.
- Whether the current name is misleading enough to justify a filename change later.
- Whether a broader `guidance field` file would add clarity or only duplication.

### Invariants
- Durable user-side continuity is real and load-bearing.
- The active guidance field is broader than this file.
- Formal role, accountability, and epistemic leadership must not be collapsed.

## Plan

### Strategy
Compare four candidates:

1. `Demote or merge`
   Fold the file into other root docs.
2. `Rename into full guidance field`
   Treat this file as the whole active guidance field.
3. `Keep unchanged`
   Leave the current file and wording as-is.
4. `Keep with minimal delta`
   Preserve the file, but clarify its scope as durable user-side continuity inside the broader guidance field.

Choose the strongest candidate under the branch-safe tests.

## Implementation Steps

  - `RATIONAL_SUBJECT.md`

Observed cost:
- overclaims the file's scope
- risks turning stored user-side continuity into the whole field
- duplicates the operational territory of:
  - `INTERACTION_PROTOCOL.md`
  - `VALUE_SIGNAL_LOOP.md`
- risks hardening provisional inferred structure as if it were constitutional

#### Candidate 3: Keep unchanged

Current benefit:
- preserves continuity surface
- avoids drift by inertia

Observed cost:
- the current wording still implies a one-sided ownership model
- the file understates the distinction between:
  - durable user-side continuity
  - the broader active guidance field
- the `Candidate Better Term` section signals the problem but does not resolve it

#### Candidate 4: Keep with minimal delta

Current benefit:
- preserves the compact continuity surface
- clarifies that this file is not the whole guidance field
- aligns the file with dynamic measure and relation-first structure
- keeps later rename options open without forcing them now

Observed cost:
- filename remains imperfect
- some residual naming tension remains visible
