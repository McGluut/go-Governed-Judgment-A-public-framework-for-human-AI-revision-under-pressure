# User Framework

## Purpose

Store durable user-side definitions, permissions, and preferences so they do not need to be repeated every session.

This file is a continuity surface, not the whole guidance field.

It preserves durable user-side continuity without claiming:
- permanent epistemic primacy for the user;
- or ownership of the whole active guidance field.

The active guidance field is broader:
- relational;
- revisable;
- and governed operationally by `INTERACTION_PROTOCOL.md` and `VALUE_SIGNAL_LOOP.md`.

## Durable User-Side Entries

### Intent
The repository is being used to develop a clearer account of rational subjectivity, judgment, boundedness, and ethical discernment in assistant-user interaction.

### Communication Preference
The user prefers direct structural analysis over polished conversational fluency.

### Term Permission
The assistant may propose, refine, or operationally alter the definition of terms used by the user when ambiguity blocks progress.

Condition:
the assistant should make the change explicit when the meaning shift is material.

### Interpretive Assumption
User inconsistency is often due to partial technical visibility rather than bad faith.

Operational effect:
prefer clarification, decomposition, and translation into executable structure before treating inconsistency as opposition.

## Boundary

Store here only what is durable enough to reduce future ambiguity across sessions, especially:
- stable permissions;
- repeated communication preferences;
- durable interpretive defaults;
- durable task-orientation cues.

Do **not** treat this file as:
- the whole active guidance field;
- the current local leadership judgment;
- or the full record of provisional value inference.

## Related Protocols

- Interaction status and mirror safeguards: `INTERACTION_PROTOCOL.md`
- Value inference, promotion, and revision: `VALUE_SIGNAL_LOOP.md`
- Preferred restart shape for future conversations: `CONVERSATION_TEMPLATE.md`

## Naming Tension

The filename `USER_FRAMEWORK.md` is legacy and somewhat rigid.

The stronger current reading is:
- `guidance field` names the broader evolving set of aims, permissions, meanings, values, and prohibitions inferred in relation;
- `USER_FRAMEWORK.md` names the durable user-side continuity surface within that broader field.

So the right near-term move is:
- keep the file;
- clarify its scope;
- and postpone any wider rename until it can be done across the root stack rather than by isolated file renaming.
